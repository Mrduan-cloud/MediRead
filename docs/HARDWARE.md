## 3. ROI 测算 — MediRead 能不能覆盖成本？

体检报告解读是典型的「B 端私有化 + C 端 SaaS」混合业务：

| 营收模式 | 单笔金额 | 覆盖单卡 4090（3w）需要 | 覆盖 A100 40G（15w）需要 |
|---|---|---|---|
| 医院 / 体检中心私有化部署 | 30–100 万 / 单 | **1 个客户就回本** | 1 个客户也覆盖 |
| 体检中心 SaaS 订阅 | 1–5 万 / 月 / 客户 | **3–6 个客户/年** | 15–30 个客户/年 |
| C 端单次解读付费 | 5–30 元 / 次 | 2,000–10,000 次 | 10,000–50,000 次 |
| 保险公司 / 健康管理公司 API | 0.5–2 元 / 次 | 50,000–150,000 次 | 250,000–800,000 次 |

**结论**：单卡 RTX 4090 对小公司来说是「**接到第一家医院 / 体检中心就回本**」的最低门槛配置。MediRead 业务对模型智商要求比 NutriCore 低（OCR 抽取出结构化指标后，解读任务相对模板化），用 9B 量化模型完全够用。

## 4. 为什么 MediRead 默认选 9B 量化 + 单卡 4090 而不是 32B + 双卡

| 选项 | 智商上限 | 显存需求 | 整机成本 | 适合场景 |
|---|---|---|---|---|
| 7B Q4 | 较低 | 4G | 单卡 4090 ~2.5w | 通用对话 |
| **9B GLM-4 Q4_K_M** | **中等** | **~6G** | **单卡 4090 ~3w** | **MediRead 体检解读 (默认)** |
| 14B Q4 | 中高 | ~9G | 单卡 4090 ~3w | 兼顾速度与质量 |
| 32B AWQ TP=2 | 较高 | ~20G/卡 | 2× 4090 ~5w | NutriCore 多 Agent 编排场景需要 |
| 70B AWQ | 高 | ~40G | 双 A100 ~25w | 投入成本 5x，对模板化任务边际收益小 |

**为什么 MediRead 不需要 32B**：
- 报告解析 Agent 主要工作是 OCR + 字段抽取 + 归一化（模板化，规则驱动）
- 医学解读 Agent 在 RAG + 多指标联合规则下，9B 模型已经能给出符合临床指南的解读
- 危急值兜底是硬编码常量表，不依赖 LLM 智商
- 单卡 4090 整机便宜近一倍，对小公司私有化交付价格更友好

**为什么不用 GLM-4-9B FP16（不量化）**：
- FP16 需要 ~18 GB 显存，单卡 4090 余量太少，跑 PaddleOCR + BGE 时容易 OOM
- Q4_K_M 是 GGUF 量化里精度最高的之一，临床场景里和 FP16 的差异在 1-2 分以内
- 体检数据解读对**确定性**要求 > 创造性，温度都设 0.2，量化损失不明显

## 5. 如果客户预算高 / 业务规模大，升级路径

```
[当前] 单卡 RTX 4090 + GLM-4-9B Q4_K_M
   ↓ 多家医院并发
[升级 1] 单卡 L20 48G / L40S 48G → 跑 GLM-4-9B FP16 或 14B Q4
   ↓ 真实临床场景需求 / 罕见病解读
[升级 2] 双卡 A100 80G → 跑 32B AWQ，质量上一个台阶
```

升级换模型时业务代码层完全不动，只调 `.env` 的 `LLM_BASE_URL` 和 `LLM_MODEL`。

## 6. 医疗数据合规对硬件选型的影响

体检报告属于**敏感个人医疗数据**，所有推理必须**本地完成**：

| 方案 | 数据是否离开本地 | 合规性 |
|---|---|---|
| **Ollama 本地 LLM ⭐** | **不离开** | ✅ HIPAA / 等保 2.0 / GDPR 友好 |
| 调用第三方云 LLM（DeepSeek / 通义 / 智谱）| 离开本地 | ❌ 医疗场景禁止 |
| 自建 vLLM 私有部署 | 不离开 | ✅ 同 Ollama |

**所以 MediRead 强烈不建议走"云 API 节省硬件成本"路线**。必须自建 LLM 服务，4090 + Ollama 是性价比最高的私有化方案。

## 7. 面试时怎么讲这套配置（话术）

> 「MediRead 处理的是用户体检报告这类敏感医疗数据，所以推理必须本地完成，不能走任何云 LLM API。我们用 Ollama 私有化部署 GLM-4-9B-Chat 的 Q4_K_M 量化版本，单卡 RTX 4090 24G 就能跑得很流畅，整机成本约 3 万。
>
> 没用更大的 32B 是做了权衡：报告解析是模板化任务（OCR + 6 字段抽取 + 归一化），医学解读靠 RAG 检索权威指南 + 多指标联合规则做兜底，9B 模型在这个场景里已经够用，临床给出的解读没出现明显劣化。
>
> 危急值（比如肿瘤标志物显著升高）不依赖 LLM 智商，走的是硬编码常量表强制兜底，这块比模型本身更可靠。
>
> 如果未来对接区域医联体或者需要罕见病解读，可以无感升级到 L40S 48G 或 A100，业务代码不动，只换 `LLM_MODEL` 配置就行。」

这套话术传达：
1. 你**理解医疗数据合规要求**（不能走云 API）
2. 你做过**模型选型权衡**（9B 够用，32B 是过度投入）
3. 你**控制了成本**（小公司能负担的 3 万整机）
4. 你**留了升级通道**（OpenAI 兼容协议）
